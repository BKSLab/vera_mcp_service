Обзор архитектуры

Система состоит из трёх независимых сервисов, каждый из которых решает свою задачу. Такое разделение позволяет развивать и масштабировать каждый сервис отдельно, а RAG Service переиспользовать в других продуктах.

Agent Service

Оркестратор диалога. Принимает запрос, управляет контекстом сессии, вызывает инструменты через MCP, формирует ответ через LLM и стримит его клиенту.

MCP Tools Server

Инструментальный слой. Прослойка между агентом и конкретными сервисами. Новый инструмент — новый файл, агент узнаёт о нём автоматически.

RAG Service

Семантический поиск по базе знаний. Гибридный поиск, reranker, хранение в Qdrant. Самостоятельный сервис, не зависит от агента.

Конверсионная логика

Незалогиненный пользователь получает доступ только к kb_search. При персональных запросах агент органично предлагает войти в аккаунт.

Конверсионная логика

Незалогиненный пользователь получает доступ только к kb_search — консультации по правам. Когда запрос требует персональных данных (избранное, поиск по локации), агент органично предлагает войти. Естественная точка регистрации без принудительных редиректов.

Схема взаимодействия
Клиент (браузер) │ │ POST /api/v1/agent/ask { session_id, user_id?, message } ▼ Next.js Proxy │ публикует задачу в очередь ▼ RabbitMQ queue: agent.requests │ payload: { session_id, user_id, message, history[] } │ retry x3, dead-letter queue при исчерпании попыток ▼ Agent Service (worker, LangGraph) │ ├── формирует контекст диалога из history[] ├── отправляет в LLM с описанием инструментов │ LLM решает вызвать kb_search │ ├── HTTP → MCP Tools Server { tool: "kb_search", args: { query } } │ │ │ └── HTTP → RAG Service POST /search │ └── гибридный поиск + reranker │ └── возвращает чанки с источниками │ ├── добавляет чанки в контекст LLM ├── LLM генерирует финальный ответ │ └── SSE stream → клиент (токены по мере генерации)
Доставка ответа клиенту
RabbitMQ — гарантирует что задача не потеряется (надёжность на входе). Retry x3, экспоненциальная задержка, dead-letter queue при исчерпании.
SSE — доставляет ответ в браузер в реальном времени (стриминг токенов).
Гибрид: клиент открывает SSE-соединение с session_id, агент при готовности стримит токены в это соединение. RabbitMQ и SSE решают разные задачи и не конкурируют.

Agent Service

Оркестратор на базе LangGraph — фреймворка с явным графом состояний. Каждый шаг агента — узел графа, переходы между ними — рёбра с условиями. Логика читается как схема, а не как магия декораторов.

Почему LangGraph

PydanticAI — high-level фреймворк, скрывает оркестрацию за декораторами. Удобно для простых агентов, но не даёт понимания что происходит внутри и сложно расширять при нестандартном поведении.

LangGraph — medium-level фреймворк на основе явного графа состояний. Каждый шаг агента — узел графа, переходы — рёбра с условиями. Видно и контролируется точно что происходит на каждом этапе.

Явный граф — логика агента читается как схема, не как магия фреймворка
Checkpointer — встроенное управление состоянием диалога между сообщениями, история хранится автоматически
Условные рёбра — агент сам решает какой инструмент вызвать и нужен ли инструмент вообще
Стриминг — нативная поддержка SSE стриминга токенов
OpenTelemetry — трейсинг интегрируется с Phoenix
Расширяемость — новый инструмент = новый узел, граф не переписывается
Граф агента (итерация 1)
START └── analyze_intent │ LLM читает запрос, решает нужен ли инструмент │ ├── needs_tool: kb_search │ └── call_kb_search │ │ вызов MCP Tools Server (HTTP) │ └── generate_with_context │ LLM формирует ответ с чанками из RAG │ → SSE stream → клиент │ └── no_tool_needed └── generate_direct LLM отвечает напрямую → SSE stream → клиент

При добавлении новых инструментов в итерации 2 граф расширяется новыми рёбрами от analyze_intent. Существующие узлы не меняются.

Состояние диалога

LangGraph checkpointer сохраняет состояние между сообщениями в рамках сессии автоматически.

class AgentState(TypedDict): session_id: str user_id: str | None # None = незалогиненный messages: list[BaseMessage] retrieved_chunks: list[str] tool_calls: list[str] # инструменты, вызванные в этом запросе

user_id определяет доступные инструменты: незалогиненный пользователь получает только kb_search.

Подключение к MCP Tools Server
from langchain_mcp_adapters.client import MultiServerMCPClient client = MultiServerMCPClient({ "vera-tools": { "url": "http://mcp-tools-server/mcp/sse", "transport": "sse" } }) tools = await client.get_tools() # LangGraph получает список инструментов автоматически
Интеграция с RabbitMQ
RabbitMQ → queue: agent.requests payload: { session_id: str, user_id: str | None, message: str, reply_to: str ← SSE connection id для доставки ответа } Retry policy: 3 попытки, экспоненциальная задержка После исчерпания: dead-letter queue + сообщение об ошибке клиенту через SSE
Системный промпт
Роль: консультант по правам людей с инвалидностью в сфере трудоустройства
Две аудитории: соискатель и работодатель
Источники ответов: только из базы знаний, не придумывать
При отсутствии информации: честно сообщить и предложить уточнить запрос
Для незалогиненных: при персональных запросах органично предложить войти в аккаунт

RAG Service

Самостоятельный сервис семантического поиска по базе знаний. Независим от агента — может подключаться к другим продуктам. Qdrant выбран вместо pgvector именно для этого: выделенная векторная БД даёт лучшую производительность и нативный гибридный поиск.

Стек
Компонент Инструмент Назначение
Vector DB Qdrant (self-hosted) Хранение эмбеддингов, гибридный поиск
Embedding модель open source, конфигурируется Векторизация запросов и чанков
Reranker BAAI/bge-reranker-base Переранжирование, запускается локально
API FastAPI HTTP-интерфейс для MCP Server и ingestion
Pipeline поиска
1
Embed query
Векторизация запроса той же моделью, которая использовалась при индексации базы знаний.
2
Hybrid search в Qdrant
Dense search — cosine similarity по эмбеддингам (top-20). Sparse search — BM25 по ключевым словам (top-20). Результаты объединяются через RRF (Reciprocal Rank Fusion).

Только векторный поиск плохо работает на точных юридических терминах — «статья 21 ФЗ-181», «квота 2%». BM25 закрывает этот пробел.
3
Reranker
Cross-encoder переоценивает топ-20 пар (запрос, чанк) совместно — принципиально точнее отдельных эмбеддингов. Возвращает топ-5 наиболее релевантных. Добавляет ~100-200 мс latency, запускается локально, GPU не нужен.
4
Возврат результатов
Чанки с метаданными (источник, аудитория, тема) передаются в MCP Tools Server, затем в Agent Service для генерации ответа.
Стратегия чанкинга

Чанкинг — критичное архитектурное решение. Плохой чанкинг не исправить хорошей моделью.

Документ (например, ФЗ-181) └── Секция (например, "Статья 21. Установление квоты") ├── Чанк 1 (300-500 токенов) └── Чанк 2 (300-500 токенов, overlap 50-100 токенов)
Чанки небольшие — для точного поиска
Секция сохраняется как метаданные — при генерации LLM получает контекст шире чанка
Нормативные акты: сохранять номера статей и пунктов как метаданные
Авторские статьи: сохранять заголовки секций как метаданные
Очистка: убрать лишние переносы, артефакты форматирования
Метаданные чанков
{ "chunk_id": "uuid", "document_id": "fz-181-art21", "source_type": "law" | "article", "source_title": "ФЗ-181, Статья 21", "audience": "seeker" | "employer" | "both", "topic": "quota" | "rights" | "dismissal" | "workplace" | ..., "date_added": "2026-06-01", "chunk_index": 3 }

Поле audience ключевое: вопрос от работодателя фильтрует чанки с employer | both, исключая чанки только для соискателей. Фильтрация по метаданным происходит до векторного поиска — быстро и точно.

Особенности поиска по юридическим текстам

Юридические тексты — сложный случай для векторного поиска: одинаковые конструкции («работодатель обязан», «в соответствии с настоящим Федеральным законом») делают эмбеддинги разных статей похожими. Применяются четыре дополнительных приёма на этапе ingestion.

Синтетические заголовки. LLM генерирует заголовок для каждого чанка, суммирующий его суть. Заголовок включается в текст перед эмбеддингом — запрос «можно ли уволить сотрудника с инвалидностью» лучше совпадёт с таким чанком.
Гипотетические вопросы. Для каждого чанка LLM генерирует 3-5 вопросов, на которые он отвечает. Вопросы индексируются как дополнительные векторы — запрос пользователя по форме ближе к вопросу, чем к тексту закона. Пример для статьи о квотах: «Какой процент квоты обязан соблюдать работодатель?», «Сколько людей с инвалидностью должно работать в организации?», «Как рассчитывается квота на трудоустройство?»
BM25 + метаданные. Точные запросы типа «статья 21 ФЗ-181» ловятся через sparse search и фильтрацию по source_title, не через векторное сходство.
Reranker как финальный судья. Cross-encoder оценивает пару (запрос + чанк) вместе — принципиально точнее отдельных эмбеддингов. Запускается локально, GPU не нужен. Добавляет ~100-200 мс latency, что для консультанта по правам приемлемо.

Ни один из этих подходов не является серебряной пулей, но все четыре вместе дают надёжный поиск по юридическому корпусу. Реализуются на этапе ingestion и не усложняют runtime поиска.

Ingestion pipeline

Запускается офлайн при добавлении новых документов. Не влияет на hot path запросов пользователя.

1
Препроцессинг
Очистка текста, извлечение структуры. Нормативные акты: сохраняются номера статей и пунктов. Авторские статьи: сохраняются заголовки секций. Убираются лишние переносы и артефакты форматирования.
2
Чанкинг
Иерархическая нарезка: документ → секции → чанки по 300-500 токенов с перекрытием 50-100 токенов. Метаданные (source, audience, topic) сохраняются для каждого чанка. LLM генерирует синтетические заголовки и гипотетические вопросы.
3
Embedding
Векторизация каждого чанка моделью эмбеддингов. Та же модель используется при поиске — важно не менять модель после индексации.
4
Upsert в Qdrant
Сохранение вектора и метаданных. Upsert позволяет обновлять существующие документы без полной переиндексации.
API эндпоинты
Метод Путь Описание
POST /search Поиск: query + фильтры → чанки с метаданными
POST /ingest Добавить документ в базу знаний
DELETE /document/{id} Удалить документ и все его чанки
GET /health Статус сервиса

MCP Tools Server

Инструментальный слой между агентом и конкретными сервисами. Реализован на FastMCP — официальной высокоуровневой обёртке из MCP SDK Anthropic. Монтируется в FastAPI как ASGI-приложение через HTTP/SSE транспорт.

Почему FastMCP
Официальный SDK — гарантированная совместимость с MCP протоколом
Декоратор @mcp.tool() с Pydantic-валидацией аргументов из коробки
LangGraph подключается через langchain-mcp-adapters нативно
HTTP транспорт (sse_app()) — правильный выбор для продакшна с Docker, в отличие от stdio

Новый инструмент — новый файл в tools/ с декоратором @mcp.tool(). Agent Service узнаёт о нём автоматически через MCP протокол — переписывать агент не нужно.

Инструменты по итерациям
Инструмент Описание Доступность Итерация
kb_search(query) Поиск по базе знаний через RAG Service Все пользователи 1
get_user_favorites(user_id) Избранные вакансии пользователя Авторизованные 2
search_vacancies(location, query) Поиск вакансий с нормализацией локации через Dadata Авторизованные 2
find_similar_vacancies(vacancy_id) Похожие вакансии через векторный поиск Авторизованные 2

Контроль доступа

Инструменты для авторизованных пользователей проверяют user_id на уровне самого инструмента. Если user_id не передан — возвращают ошибку с кодом доступа. Агент получает этот ответ и органично сообщает пользователю что нужно войти.

Пример инструмента
@mcp.tool() async def kb_search(query: str, audience: str = "both") -> dict: """ Поиск по базе знаний о правах людей с инвалидностью. audience: 'seeker' | 'employer' | 'both' """ response = await http_client.post( f"{RAG_SERVICE_URL}/search", json={"query": query, "audience": audience, "top_k": 5} ) return response.json()

Observability

Инструмент: Arize Phoenix, self-hosted в Docker. Все три сервиса экспортируют трейсы в единый Phoenix instance через OTLP. Данные хранятся локально, доступ через внутренний порт — не публичный.

Интеграция по сервисам
Agent Service — openinference-instrumentation для LLM вызовов и span-дерева
RAG Service — ручные spans для embed + search
MCP Tools Server — spans на каждый вызов инструмента
Дерево спанов на запрос
agent.request [root span] ├── rabbitmq.consume ← время в очереди ├── llm.plan ← LLM решает какой инструмент вызвать ├── mcp.tool_call: kb_search │ └── rag.search │ ├── embed_query ← время генерации эмбеддинга │ └── vector_search ← время поиска в Qdrant └── llm.generate ← финальная генерация ответа
Ключевые метрики
Метрика Где собирается Зачем
Latency каждого спана Все сервисы Найти узкое место: LLM или RAG
RAG relevance score RAG Service Насколько чанки релевантны запросу

Итерации

Система расширяется итерационно. Новый инструмент не требует переработки агента или RAG Service — только добавление файла в MCP Tools Server.

Итерация 1

Базовая консультация

Agent Service + RAG Service + MCP Tools Server с инструментом kb_search. Вера отвечает на вопросы по правам из базы знаний. Доступна всем пользователям без регистрации.

Итерация 2

Персональные инструменты

Добавление инструментов: get_user_favorites, search_vacancies (нормализация локации через Dadata), find_similar_vacancies (векторный поиск). Доступны только авторизованным пользователям.

Итерация 3+

Платформенные инструменты

Размещение резюме, отклики на вакансии, AI-матчинг работодатель и соискатель. Вера становится активным участником трудоустройства, а не только консультантом.
