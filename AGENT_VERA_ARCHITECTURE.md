## Концепция

Вера — AI-консультант по правам людей с инвалидностью в сфере трудоустройства и трудовой деятельности. Отвечает на вопросы соискателей (права, квоты, льготы, защита от увольнения) и работодателей (как трудоустроить, какие субсидии, как оборудовать рабочее место).

База знаний строится на нормативных документах (ТК РФ, ФЗ-181 и др.) и авторских статьях-систематизациях. Это уникальная ценность, которой нет ни на hh.ru, ни на Работе России.

Инструменты (тулы) расширяются итерационно — RAG является первым и основным, далее добавляются поиск вакансий, избранное, векторный матчинг.

---

## Три сервиса

### 1. Agent Service

**Роль:** оркестратор. Принимает запрос пользователя, управляет диалогом, вызывает тулы через MCP, формирует финальный ответ через LLM.

**Стек:** FastAPI, LLM через провайдера (модель конфигурируется), MCP клиент (HTTP transport).

**Ответственность:**
- Читает задачу из RabbitMQ очереди
- Формирует system prompt с инструкцией агента
- Решает какие тулы вызвать и в каком порядке
- Вызывает MCP Tools Server по HTTP
- Получает контекст, отправляет в LLM
- Стримит финальный ответ клиенту через SSE

**Не знает** о внутреннем устройстве тулов и RAG — только о их интерфейсах через MCP.

---

### 2. MCP Tools Server

**Роль:** MCP-сервер с набором инструментов. Прослойка между агентом и конкретными сервисами/данными. Транспорт — HTTP.

**Стек:** FastAPI (или любой HTTP-сервер), MCP SDK.

**Инструменты по итерациям:**

| Тул | Описание | Доступность | Итерация |
|-----|----------|-------------|----------|
| `vera_rag_kb(query)` | Поиск по базе знаний Vera через RAG Service | Все пользователи | 1 |
| `get_user_favorites(user_id)` | Избранные вакансии пользователя | Только авторизованные | 2 |
| `search_vacancies(location, query)` | Поиск вакансий с нормализацией локации через Dadata | Только авторизованные | 2 |
| `find_similar_vacancies(vacancy_id)` | Похожие вакансии через векторный поиск | Только авторизованные | 2 |

**Принцип расширения:** новый тул = новая функция в MCP Tools Server. Агент узнаёт о нём через описание инструмента — переписывать агент не нужно.

---

### 3. RAG Service

**Роль:** самостоятельный сервис для семантического поиска по базе знаний. Переиспользуемая инфраструктура — может подключаться к другим продуктам.

**Стек:** FastAPI, Qdrant (self-hosted), модель эмбеддингов, cross-encoder reranker.

**Ответственность:**
- Индексация документов: принимает текст → генерирует эмбеддинг → сохраняет в Qdrant
- Поиск: принимает запрос → гибридный поиск (dense + sparse) → reranker → возвращает релевантные чанки
- Управление корпусом: добавление/обновление/удаление документов

**База знаний:**
- Нормативные документы: ТК РФ, ФЗ-181 «О социальной защите инвалидов», подзаконные акты
- Авторские статьи: систематизации, разборы, подборки (пишет автор проекта)
- Структура чанков: документ → секции → чанки с метаданными (источник, дата, тема)

## Паттерны взаимодействия

### Основной flow — запрос пользователя

```
Клиент
  │
  │  POST /api/v1/agent/ask  { session_id, user_id?, message }
  ▼
Next.js Proxy
  │
  │  публикует задачу в очередь
  ▼
RabbitMQ → queue: agent.requests
  │         payload: { session_id, user_id, message, history[] }
  │         политика: retry x3, dead-letter queue при исчерпании
  ▼
Agent Service (worker)
  │
  ├── формирует контекст диалога из history[]
  ├── отправляет в LLM с описанием тулов
  │
  │  LLM решает вызвать vera_rag_kb("квоты трудоустройство инвалидов")
  │
  ├── HTTP → MCP Tools Server: { tool: "vera_rag_kb", args: { query: "..." } }
  │             │
  │             └── HTTP → RAG Service: POST /search { query, top_k }
  │                           └── векторный поиск
  │                           └── возвращает чанки с источниками
  │             └── возвращает чанки агенту
  │
  ├── добавляет чанки в контекст LLM
  ├── LLM генерирует финальный ответ
  │
  └── SSE stream → клиент (токены по мере генерации)
```

### Доставка ответа клиенту

**RabbitMQ** — гарантирует что задача не потеряется (надёжность на входе).

**SSE** — доставляет ответ в браузер в реальном времени (стриминг токенов).

Гибрид: клиент открывает SSE-соединение с `session_id` → агент при готовности стримит токены в это соединение. RabbitMQ и SSE решают разные задачи и не конкурируют.

### Retry policy (RabbitMQ)

- Попытки: 3
- Задержка между попытками: экспоненциальная
- После исчерпания: сообщение → dead-letter queue для разбора вручную
- Клиенту: сообщение об ошибке через SSE

## Конверсионная логика

Незалогиненный пользователь получает доступ только к `vera_rag_kb` — консультации по правам. Когда запрос требует персональных данных (избранное, поиск по локации), агент органично предлагает войти. Естественная точка регистрации без принудительных редиректов.

## Agent Service — внутреннее устройство

### Фреймворк: LangGraph

**Почему LangGraph, а не PydanticAI или другие:**

PydanticAI — high-level фреймворк, скрывает оркестрацию за декораторами. Удобно для простых агентов, но не даёт понимания что происходит внутри и сложно расширять при нестандартном поведении.

LangGraph — medium-level фреймворк на основе явного графа состояний. Каждый шаг агента — это узел графа, переходы между ними — рёбра с условиями. Ты видишь и контролируешь точно что происходит на каждом этапе. Это и есть правильное понимание архитектуры агента.

**Ключевые преимущества для этого проекта:**

- **Явный граф** — логика агента читается как схема, не как магия фреймворка
- **Checkpointer** — встроенное управление состоянием диалога между сообщениями, история хранится автоматически
- **Условные рёбра** — агент сам решает какой тул вызвать и нужен ли тул вообще
- **Стриминг** — нативная поддержка SSE стриминга токенов
- **OpenTelemetry** — трейсинг интегрируется с Phoenix
- **Расширяемость** — новый тул = новый узел, граф не переписывается

### Граф агента Веры (итерация 1)

```
START
  └── [node] analyze_intent
        │  LLM читает запрос пользователя,
        │  решает нужен ли инструмент
        │
        ├── needs_tool: vera_rag_kb
        │     └── [node] call_kb_search
        │               │  вызов MCP Tools Server (HTTP)
        │               │  получает релевантные чанки из RAG
        │               └── [node] generate_with_context
        │                         LLM формирует ответ с контекстом
        │                         → SSE stream → клиент
        │
        └── no_tool_needed
              └── [node] generate_direct
                        LLM отвечает напрямую
                        → SSE stream → клиент
```

При добавлении новых тулов (итерация 2) граф расширяется новыми рёбрами от `analyze_intent` — существующие узлы не меняются.

### Состояние диалога (checkpointer)

LangGraph checkpointer сохраняет состояние между сообщениями в рамках сессии:

```python
class AgentState(TypedDict):
    session_id: str
    user_id: str | None        # None = незалогиненный
    messages: list[BaseMessage]
    retrieved_chunks: list[str]
    tool_calls: list[str]      # какие тулы были вызваны в этом запросе
```

`user_id` определяет доступные тулы — незалогиненный получает только `vera_rag_kb`.

### Инструкция агента (system prompt)

Вера получает чёткую инструкцию:
- роль: консультант по правам инвалидов в сфере трудоустройства
- две аудитории: соискатель и работодатель
- источники ответов: только из базы знаний, не придумывать
- при отсутствии информации: честно сообщить и предложить уточнить запрос
- для незалогиненных: при персональных запросах органично предложить войти

### Интеграция с RabbitMQ

Agent Service запускается как worker, слушает очередь `agent.requests`:

```
RabbitMQ → agent.requests
  payload: {
    session_id: str,
    user_id: str | None,
    message: str,
    reply_to: str   ← SSE connection id для доставки ответа
  }
```

После обработки стримит токены через SSE соединение по `reply_to`.

## RAG Service — внутреннее устройство

### Стек

| Компонент | Инструмент | Назначение |
|-----------|-----------|------------|
| Vector DB | **Qdrant** (self-hosted) | Хранение эмбеддингов, векторный и гибридный поиск |
| Embedding модель | open source (конфигурируется) | Векторизация запросов и чанков |
| Reranker | cross-encoder (локально) | Переранжирование результатов поиска |
| API | FastAPI | HTTP интерфейс для MCP Tools Server и ingestion |

Qdrant выбран вместо pgvector потому что RAG Service — самостоятельная переиспользуемая инфраструктура для нескольких продуктов. Выделенная векторная БД даёт лучшую производительность, нативный гибридный поиск и независимость от основного PostgreSQL.

---

### Стратегия чанкинга

Чанкинг — критичное архитектурное решение. Плохой чанкинг не исправить хорошей моделью.

**Подход: иерархический чанкинг**

```
Документ (например, ФЗ-181)
  └── Секция (например, "Статья 21. Установление квоты")
        └── Чанк 1 (300-500 токенов)
        └── Чанк 2 (300-500 токенов)
```

- Чанки небольшие — для точного поиска
- Секция сохраняется как метаданные — при генерации LLM получает контекст шире чанка
- Перекрытие (overlap) между чанками — 50-100 токенов, чтобы не терять смысл на границах

**Препроцессинг документов:**
- Нормативные акты: сохранять номера статей и пунктов как метаданные
- Авторские статьи: сохранять заголовки секций как метаданные
- Очистка: убрать лишние переносы, артефакты форматирования

---

### Метаданные чанков

Метаданные позволяют фильтровать поиск до векторного сравнения — это быстро и точно.

```python
{
    "chunk_id": "uuid",
    "document_id": "fz-181-art21",
    "source_type": "law" | "article",   # нормативный акт или авторская статья
    "source_title": "ФЗ-181, Статья 21",
    "audience": "seeker" | "employer" | "both",  # аудитория
    "topic": "quota" | "rights" | "dismissal" | "workplace" | ...,
    "date_added": "2026-06-01",
    "chunk_index": 3                    # порядковый номер в документе
}
```

`audience` — ключевое поле: вопрос от работодателя фильтрует чанки с `audience: employer | both`, исключая чанки только для соискателей.

### Pipeline поиска

```
Запрос пользователя
  │
  ├── [1] Embed query — векторизация запроса той же моделью что при индексации
  │
  ├── [2] Hybrid search в Qdrant
  │       ├── Dense search  — cosine similarity по эмбеддингам (top-20)
  │       └── Sparse search — BM25 по ключевым словам (top-20)
  │       └── Fusion — объединение результатов (RRF — Reciprocal Rank Fusion)
  │
  ├── [3] Reranker — cross-encoder переоценивает топ-20 пар (запрос, чанк)
  │       └── возвращает топ-5 наиболее релевантных
  │
  └── [4] Возвращает чанки с метаданными → MCP Tools Server → Agent Service
```

**Почему гибридный поиск:**
Только векторный поиск плохо работает на точных юридических терминах — «статья 21 ФЗ-181», «квота 2%», конкретные названия льгот. BM25 закрывает этот пробел. Qdrant поддерживает гибрид нативно.

**Почему Reranker:**
Cross-encoder оценивает пару (запрос + чанк) вместе — принципиально точнее чем отдельные эмбеддинги. Добавляет ~100-200ms latency, что для консультанта по правам приемлемо — точность важнее скорости. Модели запускаются локально, GPU не нужен: `BAAI/bge-reranker-base` или аналог.

---

### Особенности поиска по юридическим текстам

Юридические тексты — худший случай для чистого векторного поиска. ТК РФ и ФЗ-181 написаны с намеренно повторяющимися конструкциями: «работодатель обязан», «в соответствии с настоящим Федеральным законом». Эмбеддинги разных статей оказываются близки в векторном пространстве просто из-за одинакового языка, а не смысла. Нумерация статей («статья 21» vs «статья 12») отличается только цифрой — модель их почти не различает.

**Решения — все применяются на этапе ingestion:**

**1. Синтетические заголовки чанков**

При индексации LLM генерирует заголовок для каждого чанка, суммирующий его суть. Заголовок включается в текст перед эмбеддингом:

```
[Заголовок]: Ограничения при увольнении работника с инвалидностью
[Текст]: Расторжение трудового договора с работниками, являющимися...
```

Запрос «можно ли уволить инвалида» лучше совпадёт с таким чанком чем с голым текстом статьи.

**2. Индексация гипотетических вопросов**

Для каждого чанка LLM генерирует 3-5 вопросов на которые он отвечает. Вопросы индексируются вместе с чанком как дополнительные векторы. Запрос пользователя по форме ближе к вопросу чем к тексту закона — это существенно улучшает recall.

Пример для статьи о квотах:
```
- Какой процент квоты обязан соблюдать работодатель?
- Сколько инвалидов должно работать в организации?
- Как рассчитывается квота на трудоустройство инвалидов?
```

**3. BM25 + метаданные для точных терминов**

Точные запросы типа «статья 21 ФЗ-181» или «квота 2%» ловятся через BM25 (sparse search) и фильтрацию по метаданным — не через векторное сходство. Именно для этого в схеме метаданных хранится `source_title` с номером статьи и названием документа.

**4. Reranker как финальный судья**

После гибридного поиска reranker переоценивает топ-20 кандидатов, рассматривая пару (запрос, чанк) вместе. Для юридических текстов это закрывает большинство ошибок первичного поиска — cross-encoder понимает смысловое соответствие значительно лучше чем косинусное сходство эмбеддингов.

**Итог:** ни один из этих подходов не является серебряной пулей, но все четыре вместе дают надёжный поиск по юридическому корпусу. Реализуются на этапе ingestion и не усложняют runtime поиска.

---

### Ingestion pipeline

Как документы попадают в базу знаний:

```
Источник (PDF / MD / TXT)
  └── [1] Препроцессинг — очистка, извлечение структуры
  └── [2] Чанкинг — нарезка с overlap, сохранение метаданных
  └── [3] Embedding — векторизация каждого чанка
  └── [4] Upsert в Qdrant — сохранение вектора + метаданных
```

Ingestion запускается офлайн при добавлении новых документов, не в hot path запросов пользователя.

API эндпоинты RAG Service:

| Метод | Путь | Описание |
|-------|------|----------|
| `POST` | `/search` | Поиск: query + фильтры → чанки |
| `POST` | `/ingest` | Добавить документ в базу знаний |
| `DELETE` | `/document/{id}` | Удалить документ и все его чанки |
| `GET` | `/health` | Статус сервиса |

## MCP Tools Server — внутреннее устройство

### Фреймворк: FastMCP (официальный MCP Python SDK)

FastMCP — высокоуровневая обёртка из официального `mcp` SDK от Anthropic. Стандарт де-факто для MCP серверов на Python. Монтируется в FastAPI как ASGI приложение через HTTP/SSE транспорт — стандартный деплой в Docker.

**Почему FastMCP:**
- официальный SDK — гарантированная совместимость с MCP протоколом
- декоратор `@mcp.tool()` с Pydantic валидацией аргументов из коробки
- LangGraph подключается через `langchain-mcp-adapters` нативно
- HTTP транспорт (`sse_app()`) — правильный выбор для продакшн с Docker, в отличие от stdio

### Структура сервиса

```python
# main.py
from fastapi import FastAPI
from mcp.server.fastmcp import FastMCP

app = FastAPI()
mcp = FastMCP("vera-tools")

app.mount("/mcp", mcp.sse_app())
```

```python
# tools/vera_rag_kb.py
@mcp.tool()
async def vera_rag_kb(query: str, audience: str = "both") -> dict:
    """
    Поиск по базе знаний о правах инвалидов.
    audience: 'seeker' | 'employer' | 'both'
    """
    response = await http_client.post(
        f"{RAG_SERVICE_URL}/search",
        json={"query": query, "audience": audience, "top_k": 5}
    )
    return response.json()
```

```python
# tools/favorites.py
@mcp.tool()
async def get_user_favorites(user_id: str) -> dict:
    """Получить избранные вакансии пользователя"""
    ...

# tools/vacancies.py
@mcp.tool()
async def search_vacancies(location: str, query: str = "") -> dict:
    """Поиск вакансий по локации в свободной форме через Dadata"""
    ...

# tools/similar.py
@mcp.tool()
async def find_similar_vacancies(vacancy_id: str) -> dict:
    """Найти похожие вакансии через векторный поиск"""
    ...
```

```
└── config.py
```

Новый тул = новый файл в `tools/` + регистрация декоратором. Agent Service узнаёт о нём автоматически через MCP протокол — переписывать агент не нужно.

### Подключение к LangGraph

Agent Service подключается к MCP Tools Server через `langchain-mcp-adapters`:

```python
from langchain_mcp_adapters.client import MultiServerMCPClient

client = MultiServerMCPClient({
    "vera-tools": {
        "url": "http://mcp-tools-server/mcp/sse",
        "transport": "sse"
    }
})

tools = await client.get_tools()  # LangGraph получает список тулов автоматически
```

### Контроль доступа к тулам

Тулы для авторизованных пользователей проверяют `user_id` на уровне самого тула — если `user_id` не передан, возвращают ошибку с кодом доступа. Агент получает этот ответ и органично сообщает пользователю что нужно войти.

## Observability — логирование и оценка агента

**Инструмент:** Arize Phoenix, self-hosted (Docker).

### Что трейсим

Каждый запрос пользователя порождает span-дерево:

```
[root span] agent.request
  ├── [span] rabbitmq.consume          ← время в очереди
  ├── [span] llm.plan                  ← LLM решает какой тул вызвать
  ├── [span] mcp.tool_call: vera_rag_kb  ← вызов тула
  │     └── [span] rag.search          ← векторный поиск внутри RAG Service
  │               ├── embed_query      ← время генерации эмбеддинга
  │               └── vector_search    ← время поиска в Qdrant
  └── [span] llm.generate              ← финальная генерация ответа
```

### Ключевые метрики

| Метрика | Где собирается | Зачем |
|---------|---------------|-------|
| Latency каждого span | все сервисы | найти узкое место (LLM или RAG) |
| RAG relevance score | RAG Service | насколько чанки релевантны запросу |
| Количество вызовов тулов на запрос | Agent Service | детектировать зацикливание |
| Токены (input/output) | Agent Service | контроль стоимости |
| Ошибки и retry | RabbitMQ + Agent | надёжность pipeline |

### Интеграция

Phoenix подключается через OpenTelemetry-совместимые SDK:

- **Agent Service** — `openinference-instrumentation` для LLM вызовов и span-дерева
- **RAG Service** — ручные spans для embed + search
- **MCP Tools Server** — spans на каждый вызов тула

Все три сервиса экспортируют трейсы в единый Phoenix instance через OTLP.

### Self-hosted деплой

Phoenix запускается отдельным контейнером в Docker Compose. Данные хранятся локально. Доступен через внутренний порт — не публичный.
